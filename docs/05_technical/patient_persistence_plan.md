# DentApp — Patient Persistence Plan

## 1. Purpose

This document defines the technical plan for moving the Patients module from frontend-only demo data to Supabase persistence.

This is a planning document, not an implementation or final SQL migration.

The goal is to define the first patient database structure, validation assumptions, RLS considerations, audit requirements, and migration draft before real patient CRUD is implemented.

---

## 2. Current Frontend-Only Patient State

The current Patients module uses local fake data only.

Current implementation:

- `src/features/patients/types.ts` defines `DemoPatient` and related demo-only types.
- `src/features/patients/demoPatients.ts` stores fake demo patient records.
- `src/pages/PatientsPage.tsx` reads directly from the fake dataset.
- `src/pages/PatientDetailPage.tsx` displays a read-only profile from fake data.
- `src/pages/PatientCreatePage.tsx` and `src/pages/PatientEditPage.tsx` render frontend-only forms.
- `src/features/patients/PatientForm.tsx` provides controlled inputs.
- `src/features/patients/patientFormValidation.ts` performs simple frontend validation.

Current behavior:

- no Supabase client is connected,
- no data is persisted,
- no localStorage persistence is used,
- no real patient data is present,
- create/edit submit only shows a demo-only message.

---

## 3. Future Supabase Goal

The future Supabase implementation should allow authorized users to:

- create patient profiles,
- search patient profiles,
- view patient profile details,
- update basic patient contact/profile data,
- archive or restore patient profiles through soft delete/status fields,
- view and update patient medical record summaries if permitted,
- create and view clinical notes if permitted,
- keep all patient data scoped by `clinic_id`,
- audit sensitive patient changes.

The first persistence implementation should keep scope narrow:

- start with patient profile data,
- add medical record summary storage,
- add clinical note structure if needed for read-only profile sections,
- avoid appointments, visits, treatment plans, payments, and documents until their scoped tasks.

---

## 4. Proposed Tables

Initial patient persistence should cover:

- `patients`
- `patient_medical_records`
- `clinical_notes`

Future related tables:

- `appointments`
- `visits`
- `treatment_plans`
- `documents`
- `patient_ledger_entries`

Future related tables should not be collapsed into `patients`. Keeping those domains separate supports permissions, audit history, reporting, and cleaner long-term data modeling.

---

## 5. Proposed `patients` Table Fields

The `patients` table should store basic patient identity, contact information, and administrative status.

Proposed fields:

| Field | Type direction | Notes |
|---|---|---|
| `id` | UUID primary key | Generated by database. |
| `clinic_id` | UUID foreign key | Required for multi-tenant access. |
| `first_name` | text | Required. |
| `last_name` | text | Required. |
| `date_of_birth` | date nullable | Optional, must not be in the future. |
| `gender` | text nullable | Future optional profile field. |
| `phone` | text | Required for MVP patient contact. |
| `email` | text nullable | Optional, validate format if provided. |
| `address` | text nullable | Future profile field. |
| `emergency_contact_name` | text nullable | Future profile field. |
| `emergency_contact_phone` | text nullable | Future profile field. |
| `preferred_contact_method` | text nullable | Future configurable field. |
| `status` | text | Initial values: `active`, `inactive`, `archived`, maybe `blocked` later. |
| `important_note` | text nullable | Administrative/visible note; not full medical record. |
| `created_by` | UUID nullable | Profile/user that created the record. |
| `updated_by` | UUID nullable | Profile/user that last updated the record. |
| `created_at` | timestamptz | Default now. |
| `updated_at` | timestamptz | Updated by trigger or app logic. |
| `deleted_at` | timestamptz nullable | Soft delete/archive support. |

Financial fields should not be stored directly on `patients` unless there is a strong reason. Unpaid balance should be derived from ledger tables later.

## 6. Proposed `patient_medical_records` Table Fields

The `patient_medical_records` table should store summary-level medical and dental record data.

Proposed fields:

| Field | Type direction | Notes |
|---|---|---|
| `id` | UUID primary key | Generated by database. |
| `clinic_id` | UUID foreign key | Required for RLS and joins. |
| `patient_id` | UUID foreign key | References `patients.id`. |
| `anamnesis_summary` | text nullable | Summary field for early MVP. |
| `allergies` | jsonb nullable | Flexible list/structure for early MVP. |
| `current_medications` | jsonb nullable | Flexible until detailed model is stable. |
| `medical_warnings` | jsonb nullable | High-visibility warnings, permission-controlled. |
| `dental_history` | text nullable | Summary for previous dental context. |
| `risk_notes` | text nullable | Clinical risk summary. |
| `created_by` | UUID nullable | Profile/user that created record. |
| `updated_by` | UUID nullable | Profile/user that last updated record. |
| `created_at` | timestamptz | Default now. |
| `updated_at` | timestamptz | Updated by trigger or app logic. |

Open model choice:

- Start with one medical record row per patient for MVP simplicity.
- Add versioning or structured sub-sections later if legal or workflow requirements demand it.

## 9. Validation Rules

Current frontend validation:

- first name is required,
- last name is required,
- phone is required,
- email is optional but must contain `@` if provided,
- date of birth is optional but cannot be in the future,
- status is required.

Future persistence validation should include:

- enforce required fields at database and service layer,
- trim leading/trailing whitespace,
- normalize empty optional fields to `null`,
- validate `status` against allowed values,
- validate email more strictly before production if needed,
- consider phone normalization later, but avoid blocking MVP on strict international phone parsing,
- prevent future date of birth in both frontend and service/database layer,
- ensure `clinic_id` is set from authenticated profile, not user input.

Potential database constraints:

- `first_name <> ''`,
- `last_name <> ''`,
- `phone <> ''`,
- `status in ('active', 'inactive', 'archived', 'blocked')`,
- `date_of_birth <= current_date`.

## 10. RLS Policy Assumptions

Initial RLS assumptions:

- every patient belongs to one `clinic_id`,
- every patient medical record belongs to one `clinic_id` and one `patient_id`,
- every clinical note belongs to one `clinic_id` and one `patient_id`,
- access requires an authenticated active profile,
- profile `clinic_id` must match row `clinic_id`,
- frontend visibility is not security enforcement.

Role assumptions:

- `owner_admin` can access all patients and related patient records in the same clinic,
- `doctor` may initially access all clinic patients unless stricter assignment is required by the pilot,
- `specialist` may require assignment-based access later,
- `assistant` may need limited warning/preparation access but should not have full record edit access by default,
- `reception_admin` may access and edit basic patient profile/contact data,
- `inventory_responsible` should not access patients by default,
- financial fields should not be placed directly in `patients` if avoidable.

RLS should eventually separate:

- patient profile access,
- medical record access,
- clinical note access,
- financial/ledger access,
- document metadata/storage access.

Assignment-based access should be deferred until assignment relationships exist through appointments, visits, or treatment plans.

## 11. Audit Log Requirements

Important patient actions should be auditable:

- patient created,
- patient profile updated,
- patient archived,
- patient restored,
- medical warning added,
- medical warning changed,
- medical warning removed,
- medical record changed,
- clinical note added,
- clinical note edited,
- clinical note archived/deleted.

Audit log should capture:

- `clinic_id`,
- acting user/profile,
- action,
- entity type,
- entity ID,
- old values where relevant,
- new values where relevant,
- timestamp,
- optional metadata.

Audit implementation options:

- application/service-layer audit writes,
- database trigger-based audit writes,
- hybrid approach.

For patient data, a hybrid approach may be safest later: service-layer context plus database-level protection for critical changes.

---

## 12. Soft Delete / Archive Approach

Recommended approach:

- use `patients.status = 'archived'` for ordinary archival workflow,
- use `patients.deleted_at` for soft delete/removal from normal queries,
- avoid hard deleting patient records in MVP,
- keep related medical records and clinical notes linked and protected,
- consider `deleted_at` on `clinical_notes` for hidden/archived notes,
- audit all archive, restore, and delete-like actions.

Open question:

- Should archived patients remain searchable by default behind a filter, or hidden from the main patient list?

## 13. Migration Draft Outline

Do not execute this outline yet.

Draft sequence:

1. Confirm prerequisite auth/profile/clinic tables exist or create them first.
2. Enable UUID generation extension if needed.
3. Create `patients` table.
4. Add `clinic_id` foreign key to `clinics`.
5. Add profile/user foreign keys for `created_by` and `updated_by` if profile table exists.
6. Add patient field constraints for required profile fields and status values.
7. Add indexes:
   - `patients(clinic_id)`,
   - `patients(clinic_id, status)`,
   - `patients(clinic_id, last_name, first_name)`,
   - optional lower-case/search indexes later.
8. Create `patient_medical_records` table.
9. Add unique constraint for one medical record per patient if using that MVP model:
   - unique `(clinic_id, patient_id)`.
10. Add indexes:
   - `patient_medical_records(clinic_id)`,
   - `patient_medical_records(patient_id)`.
11. Create `clinical_notes` table.
12. Add indexes:
   - `clinical_notes(clinic_id, patient_id)`,
   - `clinical_notes(created_at)`,
   - optional `(clinic_id, note_type)`.
13. Add `updated_at` trigger function if project-level trigger does not already exist.
14. Attach `updated_at` triggers to patient tables.
15. Enable RLS on patient tables.
16. Add initial clinic-based and role-based policies in a later policy-focused migration.
17. Add audit log table or confirm existing audit table before audit triggers/service writes.
18. Add audit behavior in a later scoped task.

Important ordering:

- profile/clinic identity structure should exist before final RLS policies are written,
- audit table should exist before audit triggers are added,
- patient persistence service should not be implemented until the migration and RLS assumptions are accepted.

## 14. Implementation Sequence

Recommended next implementation tasks:

1. Create initial Supabase migration for `patients`, `patient_medical_records`, and `clinical_notes`.
2. Create or confirm `clinics` and `profiles` foundation if not already migrated.
3. Add initial RLS helper functions for current profile, clinic, and role.
4. Add initial RLS policies for patient profile access.
5. Add audit log table or audit strategy if not already implemented.
6. Configure Supabase client and environment variables.
7. Create patient service layer, for example `src/services/patientsService.ts` or `src/features/patients/patientsService.ts`.
8. Replace direct demo data reads with a data access abstraction.
9. Add TanStack Query only when server data fetching is explicitly scoped.
10. Replace demo-only form submit with real create/update mutations.
11. Add integration tests/manual RLS checks with demo users and roles.

## 15. Open Questions

- Should the first migration include `clinics` and `profiles`, or should those be completed in a separate auth foundation task first?
- Should `patient_medical_records` be one row per patient for MVP, or support multiple record sections/versions from day one?
- Should medical warnings live only in `patient_medical_records`, or should a non-clinical `important_note` remain on `patients` for reception/admin visibility?
- Should doctors see all clinic patients in MVP, or only assigned/relevant patients?
- How should specialist assignment be modeled before specialist RLS is enforced?
- Should reception/admin be allowed to view any medical warnings, or only non-clinical profile notes?
- Should archived patients remain searchable in the main patient list?
- Should clinical notes be editable after creation, or should edits create revisions?
- Should patient field changes be audited through triggers, service-layer writes, or both?
- What exact patient record fields are required by Serbian dental record regulations?
