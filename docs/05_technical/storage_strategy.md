# DentApp — Storage Strategy

## 1. Purpose

This document defines the initial file storage strategy for DentApp.

DentApp may store sensitive patient-related documents, images, consent forms, generated PDFs, and reports.

Storage must be designed carefully from the beginning.

---

## 2. Storage Provider

Initial storage provider:

- Supabase Storage

Supabase Storage should be used for:

- patient documents,
- consent forms,
- X-rays,
- photos,
- generated PDFs,
- internal reports,
- clinic templates.

---

## 3. Core Storage Principles

Storage principles:

- patient files must not be public by default,
- access must require authentication,
- access must respect clinic_id,
- access must respect user role,
- document metadata should be stored in the database,
- file paths should be structured,
- real patient files must not be stored in GitHub,
- production storage must be backed up or recoverable,
- file paths should not contain patient names or sensitive personal data.

---

## 4. Document Metadata

Files should have metadata stored in the database table:

- documents

Possible metadata:

- document ID,
- clinic ID,
- patient ID,
- related visit ID,
- related treatment plan ID,
- document type,
- title,
- storage bucket,
- storage path,
- file name,
- mime type,
- file size,
- uploaded by,
- visibility,
- created date,
- updated date,
- deleted date.

The database record controls discoverability and business logic.

The storage object stores the actual file.

---

## 5. Suggested Buckets

Initial bucket strategy:

    patient-documents
    generated-documents
    clinic-documents

### 5.1 patient-documents

For patient-related uploaded files:

- consent forms,
- X-rays,
- photos,
- external reports,
- scanned documents.

### 5.2 generated-documents

For PDFs generated by DentApp:

- treatment plans,
- patient record summaries,
- visit summaries,
- patient ledger exports,
- doctor reports,
- inventory reports.

### 5.3 clinic-documents

For clinic-level documents:

- templates,
- internal policies,
- general forms,
- non-patient-specific documents.

---

## 6. Suggested Path Structure

Use structured paths.

Example path structure:

    clinic-{clinic_id}/patients/{patient_id}/documents/{document_id}-{filename}
    clinic-{clinic_id}/patients/{patient_id}/generated/{document_id}.pdf
    clinic-{clinic_id}/templates/{template_id}.docx

Important:

Do not include patient names in file paths.

Use IDs, not personal data.

---

## 7. File Naming Rules

Avoid sensitive file names.

Bad example:

    Milan_Petrovic_Xray_2026.pdf

Better example:

    document-{document_id}.pdf

Original file name can be stored in metadata if needed, but avoid exposing it in public URLs.

---

## 8. Access Control

Storage access should depend on:

- authenticated user,
- clinic membership,
- role,
- patient relationship if needed,
- document visibility.

Visibility examples:

- clinical,
- administrative,
- financial,
- owner_only,
- inventory,
- public_template.

Storage access must not rely only on frontend logic.

---

## 9. Patient Documents

Patient documents may include:

- X-rays,
- intraoral photos,
- signed consent forms,
- scanned records,
- external reports,
- referrals,
- treatment plan PDFs.

Access should be restricted to authorized clinical/admin users.

---

## 10. Generated Documents

Generated documents may include:

- patient record summary,
- treatment plan PDF,
- visit summary,
- patient ledger,
- daily schedule,
- commission report,
- inventory report.

Generated documents can either be:

- generated on demand,
- stored after generation.

Decision depends on legal and operational needs.

Initial recommendation:

Start with printable HTML views and generate/store PDFs later when document formats are stable.

---

## 11. Consent Forms

Consent forms may require special handling.

Potential requirements:

- template version,
- signed file,
- patient ID,
- signing date,
- uploaded/scanned by,
- related treatment plan,
- related visit,
- retention policy,
- audit history.

Consent forms should not be casually deleted.

---

## 12. X-Rays and Images

Dental images may be large and sensitive.

Need to consider:

- file size limits,
- accepted formats,
- preview support,
- download restrictions,
- storage cost,
- backup strategy,
- whether DICOM support is needed later.

DICOM is not part of MVP unless explicitly required.

---

## 13. File Upload Rules

Upload rules should include:

- allowed file types,
- maximum file size,
- required patient association for patient documents,
- required document type,
- uploader tracking,
- upload date,
- visibility setting,
- clear error messages.

Future improvement:

- virus/security scanning if available,
- image compression or thumbnails,
- document OCR if legally and technically justified.

---

## 14. Suggested Initial Allowed File Types

Initial allowed file types may include:

- PDF,
- JPG,
- JPEG,
- PNG,
- DOCX,
- XLSX.

Potential later file types:

- DICOM,
- TIFF,
- ZIP,
- STL for dental scans,
- other imaging formats.

The allowed file list must be validated with the pilot practice.

---

## 15. File Size Rules

Initial maximum file size must be decided before implementation.

Possible starting limits:

- ordinary documents: 10 MB,
- images: 20 MB,
- large X-ray/image files: configurable,
- larger files: future review.

Open question:

What are typical X-ray/photo file sizes in the pilot practice?

---

## 16. File Deletion Rules

Avoid hard deleting patient documents unless policy allows.

Preferred approach:

- soft delete metadata,
- restrict visibility,
- keep audit record,
- optionally physically delete only by owner/admin with explicit confirmation.

If physical deletion is supported, it must be restricted and audited.

---

## 17. Document Versioning

Some documents may need versioning.

Examples:

- consent form template,
- generated treatment plan,
- patient record export,
- signed documents.

MVP may start without full versioning but should not overwrite important signed or generated documents silently.

---

## 18. Audit Requirements

Audit storage-related actions:

- document uploaded,
- document viewed if required,
- document downloaded if required,
- document renamed,
- document deleted/archived,
- document visibility changed,
- generated PDF created.

For MVP, at minimum audit:

- upload,
- delete/archive,
- visibility change.

---

## 19. Backup and Recovery

Before production pilot, define:

- storage backup approach,
- restore process,
- retention period,
- who can access backups,
- incident procedure.

Production patient files must not depend only on a single unverified storage location.

---

## 20. Development and Demo Data Rules

During development:

- use fake files,
- use demo documents,
- do not upload real patient documents,
- do not commit patient documents to GitHub,
- do not use real X-rays or patient images unless properly anonymized and approved.

Demo files should be clearly fake.

---

## 21. MVP Simplification

For MVP:

- use private Supabase buckets,
- store metadata in documents table,
- use ID-based paths,
- avoid patient names in paths,
- restrict access by clinic and role,
- support upload/download for authorized users,
- support generated printable documents later,
- keep document model simple but future-ready.

---

## 22. Open Questions

- Should generated PDFs be stored or generated on demand?
- What file types should be allowed in MVP?
- What is the maximum file size?
- Are X-rays stored in app or only linked externally?
- Are signed consent forms scanned or digitally signed?
- Should document view/download be audited?
- Should files be encrypted additionally before upload?
- How will storage backups be handled?
- Should patient documents be grouped by visit, document type, or timeline?
- Should document templates be editable inside the app?
